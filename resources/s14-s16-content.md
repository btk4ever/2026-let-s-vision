# S14-S16 Content Reference

## S14 — 精简注入

副标题: Context Engineering · 只给模型它真正需要的信息

### 卡片 1: AGENTS.md 极简化
- 控制在 100 行以内
- 只写领域知识，不写通识
- 行为式引用："如果改 X → 必须读 Y"
- 渐进式披露，按需展开

### 卡片 2: 辅助定位脚本
- 模块查找 & 文件定位
- 架构信息速查
- 避免 agent 重复探索
- 示例: `$ ./scripts/find-module.sh auth` → `src/modules/auth/` → 3 files, 420 lines

### 卡片 3: 选择顶级模型
- Agent 编码是推理密集任务
- 弱模型 → 错误积累 → 返工
- 省下的 token 费不如省下的人力
- 警告: 省成本选弱模型 = 错误学习

### 演讲备注
知道了问题在哪，来看怎么应对。首先是"精简注入"——只给模型必要的信息。

（点击卡片 1）
AGENTS.md 或者 CLAUDE.md，很多人恨不得往里塞几百行。但实际上，超过 100 行之后模型就很难全部记住了。所以我们的实践是：
- 控制在 100 行以内，只写领域知识——就是你项目里独有的东西，而不是"请写好代码"这类通识
- 关键文件清单不要单纯列出来，要用行为式引用："如果你要改认证模块，必须先读 auth/README.md"
- 渐进式披露，让模型按需去找更详细的信息，而不是一次全塞进去

（点击卡片 2）
第二个是辅助脚本。Agent 在大型项目里最浪费上下文的操作就是"找东西"。它会 grep、会 find、会反复读文件，这些全都消耗上下文。所以我们会准备专门的脚本——模块查找、架构信息速查，让 agent 一步到位，不要盲目搜索。

（点击卡片 3）
第三个，选模型别省钱。Agent 编码是推理密集型任务，弱模型会产生错误代码，然后你要花时间纠正，纠正本身又消耗上下文，形成恶性循环。省下的那点 token 费用远不如省下的人力成本。

## S15 — 控制边界

副标题: Context Engineering · 隔离、拆分、保持上下文干净

### 任务拆分原则（流程图）
大型任务（跨模块/多文件）→ 拆分（明确边界与依赖）→ Task A / Task B / Task C（各 1 context）→ 合并交付

判断标准: 超过一个 context window 搞不定 = 必须拆

### 卡片 1: Sub-agent 协作
- Sub-agent / agent team 有帮助
- 但合理拆分比多 agent 更有效
- 先拆好任务，再决定是否并行

### 卡片 2: 新开 Thread
- 每个任务用干净的上下文
- 避免无关历史污染注意力
- 完成一个任务 → 开新线程继续

### 演讲备注
第二组实践：控制边界。

（点击流程图）
最重要的一条：任务拆分。我给一个非常简单的判断标准——如果一个任务超过一个 context window 还搞不定，那它就是"大型任务"，必须拆。

拆成什么样？每个子任务应该在一个 context window 内能完成。明确边界和依赖关系，分别执行，最后合并。

（点击 Sub-agent）
很多人会问 sub-agent 或者 agent team 有没有用？有用。但是它们的前提是你先把任务拆好了。如果任务本身边界不清，丢给再多 agent 也是乱的。合理的拆分比多 agent 更有效。

（点击新开 Thread）
最后一个很容易被忽略的实践：新开 thread。很多人习惯在一个对话里连续做好几个任务，但前面任务的历史会污染后面任务的注意力。完成一个任务，开一个新线程，保持干净的上下文。成本很低，效果很好。

## S16 — 模型注意力分布

### 柱状图
注意力强度（从左到右）: 90% 75% 50% 35% 25% 20% 18% 20% 25% 35% 55% 70% 85%
标签: 开头（强）— 中段（弱）— 最近（强）

### 总结
精简注入 + 控制边界 → 关键信息始终在注意力高位

### 演讲备注
为什么前面两页的实践有效？这张图就是背后的原理。

模型的注意力分布呈现一个很典型的 U 型曲线：开头和最近的内容获得强注意力，中段是衰减区。

所以我们前面做的所有事情，本质上就是：
- 精简注入：AGENTS.md 里的规则放在开头，模型一开始就能看到
- 控制边界：保持上下文短而干净，避免关键信息被挤到中段的衰减区
- 新开 thread：每次都让任务描述处于"最近"的高注意力位置

这就是上下文工程的核心——不是写更好的提示词，而是管理好信息在上下文中的位置和密度。
